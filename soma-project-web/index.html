soma-next in the browser — phase 1d

A WebAssembly build of soma-next loaded in this tab boots a real Runtime with the full DefaultPortRuntime pipeline — lifecycle gate, sandbox check, policy check, auth check, input schema validation. The three in-tab ports (dom, audio, voice) are registered on top of the bootstrap flow the same way native dylib ports are registered on a server soma-next. soma_invoke_port dispatches through DefaultPortRuntime::invoke, exactly the same code path every native proof project uses. Everything above this paragraph is static HTML — nothing below it is.

runtime

The harness boots the hello pack automatically on first load. The boot buttons let you re-boot with different manifests and inspect the resulting state.

LLM brain (phase 1g)

The brain lives outside the wasm tab. JavaScript POSTs a prompt plus the runtime's port catalog to a configurable endpoint (a thin proxy in front of Claude / GPT / local model, or a fixture for tests). The endpoint returns a plan — an ordered list of {port_id, capability_id, input} steps. This harness then invokes each step via soma_invoke_port, producing the same PortCallRecord chain an autonomous SOMA run would produce. The runtime is universal; the brain is swappable by changing one URL.

autonomous goal execution (phase 1e)

Submits a natural-language goal to the runtime's SessionController. The selector picks the say_hello skill from the hello pack, the executor binds the text input from the working-memory binding injected from the goal string, and the skill invokes dom.append_heading through the real DefaultPortRuntime. Repeated invocations feed the multistep learning pipeline — after ~5 runs the schema induction + routine compilation kicks in and future runs enter plan-following mode.
